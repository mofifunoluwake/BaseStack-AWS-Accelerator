## 1. The Big Picture (Why This Matters)

In the cloud, your perimeter is not a firewall — it is an identity. There are no physical walls between the internet and your AWS infrastructure. The only thing standing between an attacker and your EC2 instances, S3 buckets, databases, and customer data is a set of credentials: a username, a password, and ideally a second factor of authentication.

When those credentials are compromised — through phishing, credential stuffing, leaked access keys in a public GitHub repository, or brute-force attacks — everything that identity has access to is immediately accessible to the attacker. No alarm goes off. No door rattles. They walk straight in.

**The statistic:** Over 80% of hacking-related breaches involve compromised credentials. MFA reduces account compromise risk by over 99% when properly enforced.

**For Nigerian fintechs:** In 2023, multiple Nigerian fintech accounts were compromised through phishing attacks targeting staff with excessive IAM permissions. The CBN Cybersecurity Framework requires MFA for all privileged system access. The NDPC (Nigeria Data Protection Commission) data breach notification obligations make credential security a legal risk, not just a technical one.

## 2. MFA Types — Know the Differences

Not all MFA is equal. Each type has different strength, cost, and usability.

### Virtual MFA Device (Authenticator App — TOTP)

| What it is | A Time-Based One-Time Password generated by an app (Google Authenticator, Authy, Microsoft Authenticator) |
|------------|----------------------------------------------------------------------------------------------------------|
| How it works | App and AWS share a secret key. Every 30 seconds, both generate the same 6-digit code using that key + current timestamp |
| Cost | Free |
| Who should use it | Standard IAM users |
| Strength | Good — resistant to password spraying |
| Vulnerability | Can be phished in real-time (attacker intercepts the code live), but impractical for most attackers |
| Exam insight | Virtual MFA is the default recommendation for standard IAM users |

### Hardware MFA Token (Physical TOTP Device)

| What it is | A dedicated physical device (like Gemalto token) that generates TOTP codes |
|------------|---------------------------------------------------------------------------|
| Advantage over virtual | Cannot be compromised by smartphone malware. Smartphone theft with unlocked authenticator app gives both factors. Hardware token is a separate device. |
| Cost | $20–50 per device |
| Who should use it | Administrators, finance teams, root account holders — anyone whose compromise would have severe business impact |
| Strength | Higher than virtual MFA — tokens cannot be cloned or copied from a screenshot |
| Exam insight | Hardware MFA devices are more secure than virtual MFA apps |

### FIDO2 / WebAuthn Security Key (e.g., YubiKey, Google Titan)

| What it is | Uses public-key cryptography instead of shared secrets. Private key never leaves the device. |
|------------|----------------------------------------------------------------------------------------------|
| How it works | Authentication requires physical possession of the key AND interaction (pressing a button) |
| Phishing resistance | **Phishing-resistant by design** — even if tricked into authenticating on a fake site, the FIDO2 key refuses (cryptographically verifies site origin) |
| Cost | $25–70 per key |
| Who should use it | Root account, security team, executives, CI/CD pipeline service accounts |
| Strength | **Strongest MFA option available** |
| Exam insight | FIDO2/WebAuthn security keys are phishing-resistant — TOTP codes can be intercepted, FIDO2 cannot |

### MFA Comparison Summary

| Type | Strength | Phishing-resistant? | Cost | Best for |
|------|----------|---------------------|------|----------|
| Virtual MFA (TOTP app) | Good | No | Free | Standard users |
| Hardware MFA token | Higher | No | $20–50 | Admins, root account |
| FIDO2/WebAuthn key | Strongest | **Yes** | $25–70 | Root, security team, executives |

## 3. The 8 IAM Security Best Practices (Memorize These)

### Rule 1: Lock down the root account immediately

The root account has unrestricted access to everything — it cannot be limited by SCPs or IAM policies.

- Enable MFA on root account before anything else
- Store root password in a secrets vault (not your personal password manager)
- Never create access keys for root
- Never use root for daily work — create an IAM admin user instead

### Rule 2: Never create root access keys

Root access keys cannot be scoped, rotated to a specific role, or restricted by any policy. If they leak, your entire account is compromised with no recovery path.

**If your account has existing root access keys — delete them now.**

### Rule 3: Enable MFA for all human IAM users

Every human user — not just admins — should have MFA enforced. Use an IAM policy with condition key `aws:MultiFactorAuthPresent: false` to deny all actions if MFA was not used at login.

### Rule 4: Apply Principle of Least Privilege to every identity

Grant only the minimum permissions required for a task. Start with zero and add permissions as needed — never start with broad permissions and try to restrict later.

Use IAM Access Advisor to identify permissions granted but not used in the last 90 days. Remove them.

### Rule 5: Use IAM roles instead of long-term access keys

Access keys are static, long-lived credentials that can be accidentally committed to code repositories. IAM roles issue temporary credentials (via STS) that expire automatically.

For EC2, Lambda, and ECS workloads — always use instance roles and execution roles, never hardcoded access keys.

### Rule 6: Rotate access keys every 90 days — and audit with IAM Credential Report

If access keys must be used, rotate them regularly. The IAM Credential Report (a CSV from the IAM console) shows every user, their access key age, last used date, and MFA status.

Run this report in your monthly security review. Keys unused for 90 days should be disabled.

### Rule 7: Use Permission Boundaries to delegate safely

When you allow a team to create their own IAM roles (e.g., a dev team creating Lambda execution roles), use Permission Boundaries to set a hard ceiling on what those roles can include.

The boundary prevents privilege escalation — a developer cannot create a role with more permissions than their own boundary allows.

### Rule 8: Use IAM Access Analyzer to detect unintended external access

IAM Access Analyzer continuously evaluates S3 bucket policies, IAM roles, KMS keys, SQS queues, and Lambda functions to identify resources accessible from outside your account or organization.

If a bucket policy accidentally allows public access, Access Analyzer flags it within minutes.

## 4. Enforcing MFA with IAM Policies (Step by Step)

### Step 1: Enable MFA on the IAM User or Root Account

1. IAM console → Users → select user → Security Credentials tab → Assign MFA Device
2. Choose device type: virtual (app), hardware token, or FIDO2 security key
3. For virtual MFA: scan QR code with authenticator app, enter two consecutive TOTP codes to confirm
4. For root: sign in as root → Account Settings → Enable MFA

### Step 2: Write an IAM Policy That Enforces MFA

Creating the MFA device is not enough. You must also write a policy that DENIES actions if the user authenticated without MFA.

The condition key is `aws:MultiFactorAuthPresent`.

Here's the MFA enforcement policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyWithoutMFA",
      "Effect": "Deny",
      "NotAction": [
        "iam:CreateVirtualMFADevice",
        "iam:EnableMFADevice",
        "iam:GetUser",
        "sts:GetSessionToken"
      ],
      "Resource": "*",
      "Condition": {
        "BoolIfExists": {
          "aws:MultiFactorAuthPresent": "false"
        }
      }
    }
  ]
}
